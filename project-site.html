The deployment flow

From a git push to a live website — step by step.

01
Create repo & project structure
Initialize a local folder, create a src/ directory for website files and a Jenkinsfile at root. Push to GitHub.
mkdir s3-website && cd s3-website && git init
↓
02
Create & configure S3 bucket
Create bucket, enable versioning, turn on static website hosting, and apply a public-read bucket policy. Versioning means every deploy is recoverable.
aws s3 mb s3://your-bucket --region us-east-1
↓
03
Jenkins Master — connect Slave
The slave EC2 needs Java, AWS CLI, and Git. Add it to master via SSH in Jenkins → Manage Nodes. Label it slave — the Jenkinsfile targets this label.
↓
04
Store AWS credentials in Jenkins
AWS Access Key ID and Secret stored as a Jenkins credential (type: AWS). The Jenkinsfile references the credential ID — keys never touch the codebase.
Credentials ID: aws-s3-credentials
↓
05
Create Jenkins Pipeline job
New item → Pipeline. Source: SCM → your GitHub repo. Trigger: GitHub hook trigger for GITScm polling. Script path: Jenkinsfile.
↓
06
Configure GitHub Webhook
GitHub repo → Settings → Webhooks. Payload URL points to Jenkins master public IP on port 8080. Event: push only. Now every commit auto-triggers Jenkins.
http://<MASTER-IP>:8080/github-webhook/
↓
07
Push code → site is live
git push triggers the webhook → Jenkins runs 4 stages on the slave → files sync to S3 → website is updated. The full loop takes under a minute.
git push origin main ✓

What this project teaches

Skills that translate directly to real engineering teams.

🔗
Event-driven automation
Webhooks replace polling. A git push fires an HTTP POST — Jenkins reacts immediately. This is how modern CI/CD actually works.
🏗️
Master / Slave architecture
Master orchestrates, slave executes. Isolating workloads means your master stays stable and slaves can be scaled or replaced independently.
📦
Pipeline as code
The Jenkinsfile lives in the repo. Your pipeline is version-controlled, reviewable, and reproducible — not a click-ops setup hidden in a UI.
🔐
Secrets management
AWS credentials stored in Jenkins credential store, injected at runtime. Keys never appear in code, logs, or git history.
📜
S3 versioning
Every deploy creates a new object version. A bad push? Roll back in seconds. This is the foundation of safe, auditable deployments.
⚡
Static site delivery
S3 + CloudFront is how massive sites serve millions of users. No server to manage, no runtime — just files delivered from the edge.
